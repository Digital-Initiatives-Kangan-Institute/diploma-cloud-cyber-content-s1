## 1. World overview

All three S1 cloud clusters play out inside **one shared YAT College world**: a vocational college engaging
the consultancy **MTS** to move its systems to AWS. Three systems rotate through assessment and practice
roles across the clusters — the **LMS**, the public **Website**, and the **Ledgerline** accounting system —
each *assessed* in at most one cluster and reused only as *practice* elsewhere (the no-leakage invariant).
The systems advance through fixed architectural states (on-prem → single-AZ → HA), and a completed project
persists into later states. The durable model — the system↔cluster matrix, the state progressions, the CL3
"improvement not HA" framing, and the single-source-on-the-website rule — is **not restated here**; see
[scenario-flow.md](scenario-flow.md) and [website-architecture.md](website-architecture.md). This plan is the
S1 *instance*: the concrete elements that make each cluster's assessments meaningful and do-able.

The elements fall into three kinds: **environmental/access** (the labs and deployable baselines the ATs run
in), **narrative/situational** (stakeholders to role-play, baseline state, constraint documents), and
**artefact/template** (supplied templates, provided code/IaC appendices, the engagement document chain).

### SE-01 — AWS Academy lab environment
- **Kind:** environmental / access
- **Location:** AWS Academy — Cloud Foundations [104469] + Cloud Architecting [172221]; cloud vendor
  services, managed DB, serverless/IaC, console/CLI/SDK, IDE, browser, SSH/RDP. The shared compute substrate
  for every cluster's build/deploy/operate work (and CL3's team build + individual deploy/operate).
- **Satisfies:** SR-CL1-01 · SR-CL2-01 · SR-CL3-01

### SE-02 — YAT College case study & ICT baseline
